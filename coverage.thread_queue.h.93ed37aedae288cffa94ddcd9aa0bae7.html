65 * \brief Functionality for thread-safe parallel processing of queued items
66 *
67 * These functions and classes provide functionality for one or more \e
68 * source threads to feed items into a first-in first-out queue, and one or
69 * more \e sink threads to consume items. This pipeline can also extend
70 * over two queues, with one or more \e pipe threads consuming items of one
71 * type from the first queue, and feeding items of another type onto the
72 * second queue.
73 *
74 * As a graphical representation of the pipeline, the following workflows
75 * can be achieved:
76 *
77 * \code
78 * [source] \ / [sink]
79 * [source] -- queue<item> -- [sink]
80 * [source] / \ [sink]
81 * .. ..
82 * N_source N_sink
83 * \endcode
84 *
85 * or for a deeper pipeline:
86 *
87 * \code
88 * [source] \ / [pipe] \ / [sink]
89 * [source] -- queue<item1> -- [pipe] -- queue<item2> -- [sink]
90 * [source] / \ [pipe] / \ [sink]
91 * .. .. ..
92 * N_source N_pipe N_sink
93 * \endcode
94 *
95 * By default, items are push to and pulled from the queue one by one. In
96 * situations where the amount of processing per item is small, items can
97 * be sent in batches to reduce the overhead of thread management (mutex
98 * locking/unlocking, etc).

100 * The simplest way to use this functionality is via the
101 * Thread::run_queue() and associated Thread::multi() and Thread::batch()
102 * functions. In complex situations, it may be necessary to use the
103 * Thread::Queue class directly, although that should very rarely (if ever)
104 * be needed.

111 //! A first-in first-out thread-safe item queue
112 /*! This class implements a thread-safe means of pushing data items into a
113 * queue, so that they can each be processed in one or more separate
114 * threads.
115 *
116 * \note In practice, it is almost always simpler to use the convenience
117 * function Thread::run_queue(). You should never need to use the
118 * Thread::Queue directly unless you have a very unusual situation.
119 *
120 * \section thread_queue_usage Usage overview
121 *
122 * Thread::Queue has somewhat unusual usage, which consists of the following
123 * steps:
124 * - Create an instance of a Thread::Queue
125 * - Create one or more instances of the corresponding
126 * Thread::Queue::Writer class, each constructed with a reference to the
127 * queue. Each of these instances will automatically notify the queue that
128 * its corresponding thread will be writing to the queue.
129 * - Create one or more instances of the corresponding
130 * Thread::Queue::Reader class, each constructed with a reference to the
131 * queue. Each of these instances will automatically notify the queue that
132 * its corresponding thread will be reading from the queue.
133 * - Launch all threads, one per instance of Thread::Queue::Writer or
134 * Thread::Queue::Reader. Note that one of these threads can be the
135 * current thread - simply invoke the respective functor's execute()
136 * method directly once all other threads have been launched.
137 * - Within the execute() method of each thread with a
138 * Thread::Queue::Writer:
139 * - create an instance of Thread::Queue::Writer::Item, constructed from
140 * the corresponding Thread::Queue::Writer;
141 * - perform processing in a loop:
142 * - prepare the item using pointer semantics (i.e. *item or
143 * item->method());
144 * - use the write() method of this class to write to the queue;
145 * - break out of loop if write() returns \c false.
146 * - when the execute() method returns, the destructor of the
147 * Thread::Queue::Writer::Item class will notify the queue that its
148 * thread has finished writing to the queue.
149 * - Within the execute() method of each thread with a
150 * Thread::Queue::Reader:
151 * - create an instance of Thread::Queue::Reader::Item, constructed from
152 * the corresponding Thread::Queue::Reader;
153 * - perform processing in a loop:
154 * - use the read() method of this class to read the next item from the
155 * queue;
156 * - break out of the loop if read() returns \c false;
157 * - process the item using pointer semantics (i.e. *item or
158 * item->method()).
159 * - when the execute() method returns, the destructor of the
160 * Thread::Queue::Reader::Item class will notify the queue that its
161 * thread has finished reading from the queue.
162 * - If all reader threads have returned, the queue will notify all writer
163 * threads that processing should stop, by returning \c false from the next
164 * write attempt.
165 * - If all writer threads have returned and no items remain in the queue,
166 * the queue will notify all reader threads that processing should stop, by
167 * returning \c false from the next read attempt.
168 *
169 * The additional member classes are designed to be used in conjunction
170 * with the MRtrix multi-threading interface. In this system, each thread
171 * corresponds to an instance of a functor class, and its execute() method
172 * is the function that will be run within the thread (see Thread::Exec for
173 * details). For this reason:
174 * - The Thread::Queue instance is designed to be created before any of the
175 * threads.
176 * - The Thread::Queue::Writer and Thread::Queue::Reader classes are
177 * designed to be used as members of each functor, so that each functor
178 * must construct these classes from a reference to the queue within
179 * their own constructor. This ensures each thread registers their
180 * intention to read or write with the queue \e before their thread is
181 * launched.
182 * - The Thread::Queue::Writer::Item and Thread::Queue::Reader::Item
183 * classes are designed to be instantiated within each functor's
184 * execute() method. They must be constructed from a reference to a
185 * Thread::Queue::Writer or Thread::Queue::Reader respectively, ensuring
186 * no reads or write can take place without having registered with the
187 * queue. Their destructors will also unregister from the queue, ensuring
188 * that each thread unregisters as soon as the execute() method returns,
189 * and hence \e before the thread exits.
190 *
191 * The Queue class performs all memory management for the items in the
192 * queue. For this reason, the items are accessed via the Writer::Item &
193 * Reader::Item classes. This allows items to be recycled once they have
194 * been processed, reducing overheads associated with memory
195 * allocation/deallocation.
196 *
197 * \note It is important that all instances of Thread::Queue::Writer and
198 * Thread::Queue::Reader are created \e before any of the threads are
199 * launched, to avoid any race conditions at startup.
200 *
201 * The use of Thread::Queue is best illustrated with an example:
202 * \code
203 * // the type of objects that will be sent through the queue:
204 * class Item {
205 * public:
206 * ...
207 * // data members
208 * ...
209 * };
210 *
211 *
212 * // The use a typedef is recommended to help with readability (and typing!):
213 * typedef Thread::Queue<Item> MyQueue;
214 *
215 *
216 * // this class will write to the queue:
217 * class Sender {

219 * // construct the 'writer' member in the constructor:
220 * Sender (MyQueue& queue) : writer (queue) { }
221 *
222 * void execute () {

224 * MyQueue::Writer::Item item (writer);
225 * while (need_more_items()) {
226 * ...
227 * // prepare item
228 * *item = something();
229 * item->set (something_else);
230 * ...
231 * if (!item.write()) break; // break if write() returns false
232 * }
233 * }
234 *
235 * private:

237 * };
238 *
239 *
240 * // this class will read from the queue:
241 * class Receiver {

244 * Receiver (MyQueue& queue) : reader (queue) { }

248 * MyQueue::Reader::Item item (reader);

270 * Sender sender (queue);

274 * Thread::Exec sender_thread (sender);

279 * \section thread_queue_rationale Rationale for the Writer, Reader, and Item member classes
280 *
281 * The motivation for the use of additional member classes to perform the
282 * actual process of writing and reading to and from the queue is related
283 * to the need to keep track of the number of processes currently using the
284 * queue. This is essential to ensure that threads are notified when the
285 * queue is closed. This happens either when all readers have finished
286 * reading; or when all writers have finished writing and no items are left
287 * in the queue. This is complicated by the need to ensure that the various
288 * operations are called in the right order to avoid deadlocks.
289 *
290 * There are essentially 4 operations that need to take place:
291 * - registering an intention to read/write from/to the queue
292 * - launching the corresponding thread
293 * - unregistering from the queue
294 * - terminating the thread
295 *
296 * For proper multi-threaded operations, these operations must take place
297 * in the order above. Moreover, each operation must be completed for
298 * all users of the queue before any of them can perform the next
299 * operation. The use of additional member classes ensures that threads
300 * have to register their intention to read or write from the queue, and
301 * that they unregister from the queue once their processing is done.
302 *
303 * While this could have been achieved simply with the appropriate member
304 * functions (i.e. register(), unregister(), %read() & %write() methods in
305 * the main Queue class), this places a huge burden on the developer to get
306 * it right. Using these member functions reduces the chance of coding
307 * errors, and in fact reduces the total amount of code that needs to be
308 * written to use the Queue in a safe manner.
309 *
310 * The Item classes additionally simplify the memory management of the
311 * items in the queue, by preventing direct access to the underlying
312 * pointers, and ensuring the Queue itself is responsible for all
313 * allocation and deallocation of items as needed.
314 *
315 * \sa Thread::run_queue()
316 */

763 //! convenience function to set up and run a 2-stage multi-threaded pipeline.
764 /*! This function (and its 3-stage equivalent
765 * Thread::run_queue(const Source&, const Item1&, const Pipe&, const Item2&, const Sink&, size_t))
766 * simplify the process of setting up a multi-threaded processing chain
767 * that should meet most users' needs.
768 *
769 * The arguments to this function correspond to an instance of the Source,
770 * the Sink, and optionally the Pipe functors, in addition to an instance
771 * of the Items to be passed through each stage of the pipeline - these are
772 * provided purely to specify the type of object to pass through the
773 * queue(s).
774 *
775 * \section thread_run_queue_functors Functors
776 *
777 * The 3 types of functors each have a specific purpose, and corresponding
778 * requirements as described below:
779 *
780 * \par Source: the input functor
781 * The Source class must at least provide the method:
782 * \code
783 * bool operator() (Item& item);
784 * \endcode
785 * This function prepares the \a item passed to it, and should return \c
786 * true if further items need to be processed, or \c false to signal that
787 * no further items are to be sent through the queue (at which point the
788 * corresponding thread(s) will exit).
789 *
790 * \par Sink: the output functor
791 * The Sink class must at least provide the method:
792 * \code
793 * bool operator() (const Item& item);
794 * \endcode
795 * This function processes the \a item passed to it, and should return \c
796 * true when ready to process further items, or \c false to signal the end
797 * of processing (at which point the corresponding thread(s) will exit).
798 *
799 * \par Pipe: the processing functor (for 3-stage pipeline only)
800 * The Pipe class must at least provide the method:
801 * \code
802 * bool operator() (const Item1& item_in, Item2& item_out);
803 * \endcode
804 * This function processes the \a item_in passed to it, and prepares
805 * \a item_out for the next stage of the pipeline. It should return \c
806 * true if the item processed is to be sent to the next stage in the
807 * pipeline, and false if it is to be discarded - note that this is
808 * very different from the other functors, where returning false signals
809 * end of processing.
810 *
811 * \section thread_run_queue_example Simple example
812 *
813 * This is a simple demo application that generates a linear sequence of
814 * numbers and sums them up:
815 *
816 * \code
817 * const size_t max_count;
818 *
819 * // the functor that will generate the items:
820 * class Source {
821 * public:
822 * Source () : count (0) { }
823 * bool operator() (size_t& item) {
824 * item = count++;
825 * return count < max_count; // stop when max_count is reached
826 * }
827 * };
828 *
829 * // the functor that will consume the items:
830 * class Sink {
831 * public:
832 * Sink (size_t& total) :
833 * grand_total (grand_total),
834 * total (0) { }
835 * ~Sink () { // update grand_total in destructor
836 * grand_total += total;
837 * }
838 * bool operator() (const size_t& item) {
839 * total += item;
840 * return true;
841 * }
842 * protected:
843 * size_t& grand_total;
844 * };
845 *
846 * void run ()
847 * {
848 * size_t grand_total = 0;
849 * Source source;
850 * Sink sink (grand_total);
851 *
852 * // run a single-source => single-sink pipeline:
853 * Thread::run_queue (source, size_t(), sink);
854 * }
855 * \endcode
856 *
857 * \section thread_run_queue_multi Parallel execution of functors
858 *
859 * If a functor is to be run over multiple parallel threads of execution,
860 * it should be wrapped in a call to Thread::multi() before being passed
861 * to the Thread::run_queue() functions. The Thread::run_queue() functions
862 * will then create additional instances of the relevant functor using its
863 * copy constructor; care should therefore be taken to ensure that the
864 * functor's copy constructor behaves appropriately.
865 *
866 * For example, using the code above:
867 *
868 * \code
869 * ...
870 *
871 * void run ()
872 * {
873 * ...
874 *
875 * // run a single-source => multi-sink pipeline:
876 * Thread::run_queue (source, size_t(), Thread::multi (sink));
877 * }
878 * \endcode
879 *
880 * For the functor that is being multi-threaded, the default number of
881 * threads instantiated will depend on the "NumberOfThreads" entry in the
882 * MRtrix confugration file, or can be set at the command-line using the
883 * -nthreads option. This number can also be set as additional optional
884 * argument to Thread::multi().
885 *
886 * Note that any functor can be parallelised in this way. In the example
887 * above, the Source functor could have been wrapped in Thread::multi()
888 * instead if this was the behaviour required:
889 *
890 * \code
891 * ...
892 *
893 * void run ()
894 * {
895 * ...
896 *
897 * // run a multi-source => single-sink pipeline:
898 * Thread::run_queue (Thread::multi (source), size_t(), sink);
899 * }
900 * \endcode
901 *
902 *
903 * \section thread_run_queue_batch Batching items
904 *
905 * In cases where the amount of processing per item is small, the overhead
906 * of managing the concurrent access to the various queues from all the
907 * threads may become prohibitive (see \ref multithreading for details). In
908 * this case, it is a good idea to process the items in batches, which
909 * drastically reduces the number of accesses to the queue. This can be
910 * done by wrapping the items in a call to Thread::batch():
911 *
912 * \code
913 * ...
914 *
915 * void run ()
916 * {
917 * ...
918 *
919 * // run a single-source => multi-sink pipeline on batches of size_t items:
920 * Thread::run_queue (source, Thread::batch (size_t()), Thread::multi (sink));

922 * \endcode
923 *
924 * By default, batches consist of MRTRIX_QUEUE_DEFAULT_BATCH_SIZE items

928 * \code
929 * ...
930 *
931 * void run ()
932 * {
933 * ...

938 * \endcode

940 * Obviously, Thread::multi() and Thread::batch() can be used in any

942 */
